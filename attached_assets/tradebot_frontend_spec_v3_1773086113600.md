# Техническое задание (ТЗ)

## Веб‑панель для сервиса анализа на исторических данных

Дата: 2026

------------------------------------------------------------------------

# 1. Цель проекта

Создать веб‑панель, в которой пользователь может:

1.  Авторизоваться
2.  Выбрать сервис **«Анализ на исторических данных»**
3.  Собрать конфигурацию `StartRunRequest`
4.  Запустить прогон
5.  Отслеживать статус и просматривать логи

Система работает через **REST API главного сервиса**, который
взаимодействует с внутренними **gRPC микросервисами**.

------------------------------------------------------------------------

# 2. Общая архитектура

## Backend

Архитектура:

Frontend (Web Panel) ↓ REST API (Главный сервис / BFF) ↓ gRPC ↓
Микросервисы обработки

### Аутентификация

Используется **cookie session**.

Flow:

POST /auth/login → сервер создаёт session_id → сохраняет в Redis / БД →
возвращает cookie

Cookie параметры:

-   HttpOnly
-   Secure
-   SameSite=Lax
-   Path=/

Все запросы к API выполняются с cookie.

Middleware:

RequireAuth - читает cookie - валидирует сессию - кладёт user_id и role
в context

ВАЖНО:

UserID из фронта **не является источником истины**. Бекенд определяет
пользователя по сессии.

# 9. Администрирование (MVP: приглашения/создание аккаунтов)

## 9.1 Роли и доступ
- В системе есть роли: `user`, `admin`.
- Раздел **Admin** доступен только пользователям с ролью `admin`.
- Проверка прав выполняется на бэкенде (middleware), фронт только скрывает пункты меню для не-админов.

## 9.2 Задача админа в MVP
Админ выполняет только операции онбординга пользователей:
1) **Создать аккаунт напрямую** (email + временный пароль)  
или/и  
2) **Сгенерировать код приглашения**, по которому пользователь сам создаёт аккаунт.

Рекомендуемый путь: **коды приглашения**, чтобы не передавать пароли вручную.

## 9.3 Экраны (Frontend)

### Экран A — Admin → Invitations
URL: `/admin/invitations`

Функции:
- Список активных приглашений (опционально)
- Кнопка **«Сгенерировать приглашение»**
- Отображение результата:
  - `invite_code` (с кнопкой «Copy»)
  - срок действия (`expires_at`)
  - ограничения (опционально): максимальное число активаций, роль по умолчанию

Поля формы генерации:
- `expires_in` (select: 1h / 24h / 7d / 30d)
- `max_uses` (number, по умолчанию 1)
- `role` (select: user/admin; по умолчанию user) — если хочешь выдавать админов тоже через инвайты

API (ожидается бэкендом):
- `POST /admin/invitations`
- `GET /admin/invitations` (опционально)
- `DELETE /admin/invitations/{id}` (опционально, revoke)

Пример ответа `POST /admin/invitations`:
```json
{
  "id": "inv_123",
  "invite_code": "ABCD-EFGH-IJKL",
  "expires_at": "2026-04-01T00:00:00Z",
  "max_uses": 1,
  "uses": 0,
  "role": "user"
}
```

### Экран B — Signup (по приглашению)
URL: `/signup`

Форма:
- `email`
- `password`
- `invite_code` (поле или автозаполнение из URL `?code=...`)

Поведение:
- `POST /auth/signup` с `{ email, password, invite_code }`
- при успехе: сразу логин (сервер ставит cookie) и переход в приложение
- при ошибке: показать причину (невалидный код / истёк / превышено число использований)

API:
- `POST /auth/signup`

Пример ошибок:
- `400 invite_invalid`
- `400 invite_expired`
- `400 invite_used_up`

## 9.4 Альтернатива (если нужно создавать аккаунты руками)
### Экран C — Admin → Create User (optional)
URL: `/admin/users/create`

Форма:
- email

- временный пароль (генерируется сервером или вводится админом)

API:

В этом варианте рекомендуется требовать от пользователя смену пароля при первом входе.

## 9.5 Нефункциональные требования для инвайтов (Backend expectations)
- Код приглашения должен быть:
  - достаточно длинным/случайным (не перебираемым)
  - с TTL (`expires_at`)
  - с лимитом использований (`max_uses`)
- При активации инвайта операция должна быть атомарной (защита от гонок).
- Логировать: кто создал инвайт, когда, сколько активаций.

# 8. Definition of Done

Система считается готовой когда:

-   работает login
-   загружаются справочники
-   можно создать StartRunRequest
-   можно добавить несколько detectors
-   можно запустить прогон
-   отображается статус прогона
-   доступны логи прогона


## Экран 2 — Исторический анализ: список прогонов + создание

Цель: один основной экран, на котором всегда виден **список всех прогонов** (AnalysisRun) и доступно создание нового прогона.

### 2.1 UI: таблица прогонов (AnalysisRun)
Таблица показывает все прогоны текущего пользователя (в админском режиме — все, если будет добавлено).

Колонки = ключевые поля `AnalysisRun` (точный состав зависит от модели на бэке, но минимум):

- `run_id`
- `created_at`
- `market.exchange`
- `market.category`
- `market.symbol`
- `interval`
- `from`
- `to`
- `status` (queued/running/done/failed)
- **статистика** (сводные метрики из AnalysisRun: например pnl, winrate, trades_count, max_drawdown — список уточняется под твою модель)

Действия в строке (actions):
- **Открыть** (переход на страницу деталей/логов прогона)
- **Скачать** (экспорт данных/конфига/результата)
- **Удалить** (с подтверждением)

Кнопка создания:
- Floating button **«+ Новый»** (в правом нижнем углу) или кнопка в шапке **«Создать»**.

Рекомендации UX:
- сортировка по `created_at` (по умолчанию desc)
- фильтр по `status`
- поиск по `symbol`
- пагинация (если много прогонов)

API:
- `GET /runs?limit=50&cursor=...&status=...&q=...` (параметры по необходимости)

### 2.2 Создание прогона (встроено в экран 2)
Экран создания (`StartRunRequest`) **не отдельная страница**, а открывается поверх списка:

Варианты реализации (любой приемлем):
1) **Modal** (для коротких форм)  
2) **Drawer / Side panel** (рекомендуется: лучше для длинной формы)  
3) Переход на `/runs/new`, но визуально как overlay (с сохранением таблицы на фоне)

Требование:
- пользователь всегда может закрыть форму и вернуться к списку без потери данных (черновик в localStorage).

Секции формы и поведение такие же, как в прежнем «Экран 3» (см. ниже раздел “Форма StartRunRequest”).

### 2.3 Скачать / экспорт
Кнопка **«Скачать»** в строке таблицы должна позволять минимум:
- скачать **конфиг запроса** (StartRunRequest) как JSON
- скачать **результат/статистику** (AnalysisRun) как JSON (или CSV — опционально)

API (ожидания к бэку):
- `GET /runs/{id}/request` → StartRunRequest (json)
- `GET /runs/{id}` → AnalysisRun (json)
- (опционально) `GET /runs/{id}/export?format=csv|json`

UI:
- меню выбора формата (dropdown) или одна кнопка “JSON” на MVP.

### 2.4 Удаление прогона
Кнопка **«Удалить»** в строке:
- подтверждение (dialog): «Удалить прогон? Действие необратимо»
- после успеха: строка исчезает, показывается toast

API:
- `DELETE /runs/{id}`

## Форма StartRunRequest (используется в Экран 2 → “+ Новый”)

Форма конфигурации `StartRunRequest` открывается модально/в drawer поверх таблицы.

### Market
- Exchange (select) — `GET /meta/exchanges`
- Category (select, зависит от exchange) — `GET /meta/categories?exchange=...`
- Symbol:
  - autocomplete + поиск — `GET /meta/symbols?exchange=...&category=...&q=...&limit=20`
  - всегда разрешён ручной ввод

### Data параметры
- Interval (select)
- PriceType (select)

### Период
- From / To — `datetime-local`
- валидация: from < to

### Fees (optional)
- toggle “Указать комиссии”
- если выключен: `fees = null`

### Detectors (массив)
- “Добавить детектор”
- каждый детектор:
  - Code (select) — `GET /meta/detectors`
  - Label (optional)
  - Opts (dynamic schema / raw-json fallback)

Opts (динамика):
- `GET /meta/detectors/{code}/schema`
- если schema недоступна: textarea JSON

Кнопка “Запустить”:
- `POST /runs/start`
- после успеха:
  - закрыть форму
  - обновить список прогонов (таблица)
  - опционально: авто-открыть run details
